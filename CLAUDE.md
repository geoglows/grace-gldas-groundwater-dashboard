# grace-groundwater-dashboard

## Project Overview
- Vanilla JS + rolldown-vite (`npm:rolldown-vite@7.2.5`) — single-page dashboard for visualizing GRACE/GRACE-FO regional groundwater storage anomalies
- ArcGIS Maps SDK + Calcite Components for the map UI; Plotly for time-series; zarrita for fetching GRACE Zarr stores; native `<dialog>` for modals
- Authenticates against Supabase Auth via `@aquaveo/geoglows-auth/core` (no React in this app — it consumes the lib's vanilla `core` surface, not `react`). Cross-app SSO with apps.geoglows + aquiferx is automatic via the shared Supabase project + same origin (when proxied through the portal at `/grace-groundwater/`)
- Hosted on Vercel as the `grace-groundwater-dashboard` project under `gromero-1273s-projects` team. Production is served via the portal rewrite — direct access at `https://grace-groundwater-dashboard.vercel.app/` returns a white page by design

## Architecture
- **Sub-app of the GEOGloWS portal**: integrated via three rewrite rules in `apps.geoglows/vercel.json` (`/grace-groundwater`, `/grace-groundwater/`, `/grace-groundwater/:path+`). Cataloged as `ggst` in `apps.geoglows/apps.json`
- **Three deployment contexts** handled by `vite.config.js`:
  - **Local dev** — `npm run dev`, assets at `/assets/...`
  - **Vercel preview** — autogenerated subdomain, `isVercelNonProd` guard forces `base: '/'`, assets at `/assets/...`
  - **Production via portal** — `VITE_BASE_PATH=/grace-groundwater/` produces `/grace-groundwater/assets/...` so portal-proxied URLs resolve
- **Auth listener ordering is load-bearing**: `src/auth-bootstrap.js` registers `supabase.auth.onAuthStateChange("INITIAL_SESSION", ...)` immediately on module load. `src/main.js` imports it FIRST so the listener exists before grace's zarr top-level awaits begin (which take seconds — INITIAL_SESSION would fire while awaiting and we'd miss it). The lib's 2s safety-net timeout backstops the listener
- **Surgical auth-slot rendering**: `renderAuthAction(state)` returns an HTML string; only the `<div id="auth-action">` slot's `innerHTML` is replaced on auth-state changes. The surrounding `.nav-bar` and `arcgis-map` are NEVER re-rendered — ArcGIS map components have lifecycles that must not be torn down

## Key Files
- `index.html` — single-page entry; `<div id="auth-action" class="nav-auth">` slot is at the right end of `.nav-bar` for the lib's `renderAuthAction` output
- `src/main.js` — imports `./auth-bootstrap.js` as the **first line** (auth listener must register before zarr awaits start at line ~110); then ArcGIS components, zarr stores, plot rendering
- `src/auth-bootstrap.js` — Supabase client + auth adapter + `mountSignInModal` + `#auth-action` slot updater. Module-level `authState` + surgical innerHTML
- `src/cells.js` / `src/db.js` / `src/helpers.js` / `src/polygonUploads.js` — domain logic for grid cells, IndexedDB caching, plot helpers, GeoJSON polygon uploads
- `src/style.css` / `src/modal.css` — plain CSS; modal centering uses explicit `position: fixed; transform`
- `vite.config.js` — function-with-env signature; `loadEnv` + `isVercelNonProd` guard; preserves `define: { global: 'globalThis' }` and `optimizeDeps` (required for the ArcGIS module's `global` references)
- `vercel.json` — `installCommand: "npm install --legacy-peer-deps"` (React 19 peer deps in the lib are ignored), no rewrites
- `.npmrc` — `legacy-peer-deps=true` so local `npm install` matches Vercel
- `data_preprocessors/` — out-of-band Python tooling for generating Zarr stores; not bundled

## Conventions
- Vanilla JS only (no TypeScript, no JSX, no React)
- HTML escape discipline: any `${value}` interpolation in template-string-then-innerHTML rendering MUST go through `escapeHtml()` from `@aquaveo/geoglows-auth/core`. Currently the only HTML rendering grace owns is the auth-slot, which uses the lib's pre-escaped output
- The lib owns the stable element IDs: `#geoglowsSignIn`, `#geoglowsSignOut`, `#geoglowsAuthActionAvatar`. Bind to those exact IDs in event handlers
- `mountSignInModal` is mounted ONCE at module load and bridged via the `geoglows:sign-in-requested` window event so any UI surface can request sign-in without holding a direct modal reference
- The standalone Vercel deploy (`grace-groundwater-dashboard.vercel.app`) is intentionally broken — assets reference `/grace-groundwater/assets/...` because `VITE_BASE_PATH` is set on Production. Direct-URL access is not a supported entry point; always go through the portal

## Commands
- `npm install --legacy-peer-deps` — install deps (the `--legacy-peer-deps` is also defaulted via `.npmrc`)
- `npm run dev` — Vite dev server (rolldown-vite under the hood)
- `npm run build` — production build. For the production-styled build use `VITE_BASE_PATH=/grace-groundwater/ npm run build`
- `npm run preview` — serve the production build locally

## Environment
- `VITE_SUPABASE_URL`, `VITE_SUPABASE_PUBLISHABLE_KEY` — required for any environment that signs users in. Set per-environment on Vercel (Production + Preview + Development). Setting only Production silently breaks Preview deploys
- `VITE_BASE_PATH` — set to `/grace-groundwater/` on Vercel **Production only**. Preview is handled by the `isVercelNonProd` guard, which forces `base: '/'` regardless of this var. Local dev leaves it unset (also `'/'`)
- The Supabase URL + publishable key are shared across apps.geoglows + aquiferx + grace — same project, same keys. Cross-app SSO falls out of having all three apps on the same origin (after the portal rewrites) reading the same `localStorage` token
